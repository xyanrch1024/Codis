# Codis Server 多线程模型

## 线程拓扑

```
┌──────────────────────────────────────────────────────────────────┐
│                        Server 进程                                │
├──────────────────────────────────────────────────────────────────┤
│                                                                  │
│  主线程 (1)                                                       │
│  └─ server.listen("127.0.0.1", port) 阻塞, cpp-httplib 内部 I/O   │
│                                                                  │
│  HTTP 工作线程 (N, cpp-httplib 按请求创建)                          │
│  ├─ handle_health/info/chat → 同步返回 JSON                       │
│  ├─ handle_acp()             → fire-and-forget, detach LLM         │
│  └─ handle_acp_stream()      → chunked_content_provider 回调       │
│                                                                  │
│  SSE 推送线程 (N, 同 HTTP 线程)                                    │
│  └─ chunked_content_provider → queue.pop() → sink.write()        │
│                                                                  │
│  LLM Worker 线程 (M, detach)                                      │
│  └─ run_acp_loop_broadcast() → LLM stream → EventBus::publish()   │
│                                                                  │
│  信号线程 (1)                                                      │
│  └─ asio::signal_set → SIGINT/SIGTERM → server.stop()           │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

## 线程职责

### 1. 主线程
- 阻塞在 `httplib::Server::listen()`
- cpp-httplib 内部使用 epoll/select 接受新连接
- 每个 HTTP 请求分配工作线程

### 2. HTTP 工作线程
- `handle_health()` / `handle_info()`: 同步返回 JSON
- `handle_chat()`: 同步调用 LLM，返回完整结果
- `handle_acp()`: fire-and-forget
  - 保存 user 消息到 SQLite
  - 检查 `active_sessions_[id].processing` 
  - `detach` LLM worker 线程
  - 返回 202 Accepted
- `handle_acp_stream()`: SSE 长连接
  - 订阅 `EventBus` topic
  - 推历史消息
  - `chunked_content_provider` 回调阻塞等待帧

### 3. SSE 推送线程
- 由 cpp-httplib 在 `chunked_content_provider` 回调中调用
- 阻塞在 `queue->pop()`（condition_variable.wait）
- 收到帧 → `sink.write()` → HTTP chunk → 客户端
- 客户端断开或 empty frame → `sink.done()` + 取消 EventBus 订阅

### 4. LLM Worker 线程 (detach)
- `run_acp_loop_broadcast()`: 最多 10 轮 tool call
- 调用 `OpenAICompatibleProvider::stream_chat()`（同步阻塞 HTTP）
- 每收到一个 token → `EventBus::publish(topic, frame)`
- tool 执行完成 → `EventBus::publish(topic, result)`
- 循环结束 → `EventBus::publish(topic, done)` → `active_sessions_[id].processing = false`

### 5. 信号线程
- `asio::signal_set` 异步等待 SIGINT/SIGTERM
- 收到信号 → `server.stop()` + `ioc.stop()`

## 锁模型

| 资源 | 锁类型 | 操作 | 竞争 |
|------|--------|------|------|
| **EventBus** | `shared_mutex` | publish: `shared_lock` (多读者并发)<br>subscribe/unsubscribe: `unique_lock` | 低 |
| **SseFrameQueue** | `mutex` + `condition_variable` | push: lock → queue.push → cv.notify<br>pop: lock → cv.wait → queue.pop | 低 |
| **SessionStore** (SQLite) | `mutex` | 所有 SQL 操作串行化 | 中 |
| **active_sessions_** | `shared_mutex` | processing 检查/修改 | 极低 |
| **Logger** | `mutex` | cerr + file 串行写入 | 低 |
| **ProviderRegistry** | `shared_mutex` | 读多写少 | 极低 |
| **ToolRegistry** | `shared_mutex` | 读多写少 | 极低 |

## 数据流

```
HTTP 线程 (handle_acp)
  │
  ├─ SessionStore::append_message()        ← mutex (SQLite)
  ├─ active_sessions_[id].processing       ← unique_lock
  │   └─ exchange(true) → 返回 true (已有 LLM) → 放弃
  │   └─ exchange(true) → 返回 false        → detach
  │
  └─ std::thread([...] { ... }).detach()
        │
        ▼
  LLM Worker 线程 (run_acp_loop_broadcast)
  │
  ├─ SystemContext::build_baseline()       ← SessionStore (mutex)
  ├─ OpenAICompatibleProvider::stream_chat()
  │   └─ LLMHttpClient::stream_post()      ← HTTPS POST (同步阻塞)
  │       └─ on_token → event_bus_.publish() ← shared_lock
  │           │
  │           └─ for handler in handlers:
  │                 handler(frame)
  │                   └─ queue->push(frame) ← mutex + cv.notify_one
  │
  ├─ ToolRegistry::execute() → event_bus_.publish(result)
  └─ event_bus_.publish(done)
        └─ active_sessions_[id].processing = false
              │
              ▼
  SSE 推送线程 (chunked_content_provider 回调)
  │
  └─ queue->pop()                          ← cv.wait (阻塞)
      └─ sink.write(data)                  ← HTTP chunk → Client
      └─ sink.done()                       ← EOF 或 done frame
          └─ event_bus_.unsubscribe()      ← unique_lock
```

## 关键设计决策

### 1. EventBus 而非直接操作 list
LLM Worker 不直接操作 `active_sessions_` 列表。它只 `publish` 事件，EventBus 负责分发给所有订阅者。无论 1 个还是 100 个 SSE 客户端，LLM 代码不变。

### 2. shared_mutex 而非 mutex
EventBus 的 publish 只读不写（遍历 handler），用 `shared_lock` 允许多个 LLM worker 同时 publish 到不同 topic，无相互阻塞。

### 3. detach 而非 join
LLM 可能耗时 10+ 秒，HTTP 线程必须立即返回（202）。不做 join，不做 `future.get()`。

### 4. SseFrameQueue 阻塞队列
SSE 推送端 `pop()` 用 `cv.wait()` 阻塞，零 CPU 占用。`push()` 时 `cv.notify_one()` 即时唤醒。

### 5. processing 原子旗标
`atomic<bool> processing` + `exchange(true)` 保证只有第一个请求触发 LLM，后续请求的 user 消息会在下一轮被处理。防止同 session 并发调用 LLM 导致重复收费。
